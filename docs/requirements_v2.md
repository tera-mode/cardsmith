## 1. プロジェクト全体像

### 1.1 何を作るか

既存 cardsmith MVP（6×6盤面・固定13カード・ランダムAI）に対して、以下を「一気に」追加する。

- プレイヤーレベル・経験値・通貨（ルーン）
- ストーリー / クエスト / チュートリアル
- カードコレクション・複数デッキ管理
- マテリアル（カードを構成する素材）の概念
- マテリアル抽出（カード分解）・マテリアル購入
- ガチャ（召喚）
- カード生成（鍛冶。マテリアルからオリジナルカード作成）

### 1.2 ゲームのコアループ

```
[ストーリー / 対戦] → 報酬（EXP・ルーン・カード）獲得
        ↓
[コレクション] でカード確認 → [抽出] でマテリアル化
        ↓
[ショップ] でルーン → マテリアル購入
        ↓
[鍛冶] でマテリアル → オリジナルカード生成
        ↓
[デッキ編集] で生成カード採用
        ↓
[ストーリー / 対戦] へ戻る（強くなったデッキで挑む）
```

このループの**毎ステップでプレイヤーが「次に何をすればいいか分かる」状態**を維持することが UX の最重要原則。

### 1.3 ゲームタイトルとの関係

タイトル「最強カード鍛冶師 / THE APEX CARDSMITH」の核は **§6.8 カード生成（鍛冶）**。すべての機能はカード生成体験を成立させるための前後関係として設計する。

## 2. 実装フェーズと依存関係

### 2.1 依存関係マップ

```
[基盤層]
  A. プロフィール（Lv/EXP/ルーン）─┐
  K. データ永続化  ───────────────┤
  L. ホーム画面・ヘッダー  ────────┘
                                    ↓
[コレクション層]
  D. カードコレクション
  E. マテリアル定義・コレクション
  J. デッキ管理
                                    ↓
[経済層]
  H. ガチャ
  G. マテリアルショップ
  F. マテリアル抽出
                                    ↓
[育成層]
  I. カード生成（鍛冶）
                                    ↓
[体験層]
  B. ストーリー / クエスト / チュートリアル
  C. 報酬システム（既存対戦への組み込み含む）
                                    ↓
[統合・調整]
  プレイテスト・バランス調整
```

**重要：**この順番で実装すること。下流の機能は上流の機能に依存している。

### 2.2 フェーズ分割

| Phase | 内容 | 想定日数 | 完了基準 |
|---|---|---|---|
| **P1** | 基盤：型・Firestore スキーマ拡張、プロフィール、ヘッダー、ホーム画面 | 2〜3日 | ホーム画面が表示され、Lv/EXP/ルーンが Firestore に保存される |
| **P2** | コレクション：カードコレクション、マテリアル定義、デッキ管理 | 3〜4日 | 既存13カードを所持カードとして表示、複数デッキを保存・編集できる |
| **P3** | 経済：ガチャ、マテリアル抽出、マテリアルショップ | 3〜4日 | ルーン消費でガチャ／購入ができ、抽出でマテリアルが手に入る |
| **P4** | 育成：カード生成（鍛冶） | 2〜3日 | レベル制限内でマテリアルからオリジナルカードを生成できる |
| **P5** | 体験：チュートリアル、クエスト、ストーリー演出、報酬付与 | 4〜5日 | 初回ログインでチュートリアル開始、5本以上のクエストがクリアできる |
| **P6** | 統合：既存対戦への組み込み、デッキ選択、報酬演出、プレイテスト | 2〜3日 | 既存対戦勝利で EXP/ルーンが得られる。第三者によるプレイテスト完了 |

**合計見積もり：16〜22日**

### 2.3 並行作業を避ける理由

各フェーズの完了基準を満たすまで次のフェーズに進まないこと。例えば P2 のデッキ管理が未完成のまま P5 のクエスト実装に着手すると、「報酬で得たカードをデッキに入れる」という基本動線がテストできない。

## 3. データモデル拡張

### 3.1 型定義の追加（`src/lib/types/game.ts` 拡張、または `src/lib/types/meta.ts` 新規作成）

```typescript
// プレイヤープロフィール
interface PlayerProfile {
  userId: string;
  level: number;          // 1始まり
  exp: number;            // 累積 EXP（レベルアップ判定に使用）
  runes: number;          // 通貨
  createdAt: number;
  updatedAt: number;
}

// レアリティ
type Rarity = 'C' | 'R' | 'SR' | 'SSR';

// 既存 Card 型を拡張
interface Card {
  // 既存フィールド（id, name, movement, attackRange, atk, hp, skill, cost）
  rarity: Rarity;
  origin: 'preset' | 'crafted';   // 既存13カード = preset, プレイヤー生成 = crafted
  craftedBy?: string;              // crafted の場合の作成者 userId
  craftedName?: string;            // プレイヤーが命名した名前（最大15文字）
  iconKey?: string;                // 表示アイコンの識別子
}

// 所持カード（コレクション）
interface OwnedCard {
  cardId: string;        // preset カード ID または crafted カードの instanceId
  count: number;
  isCrafted: boolean;
  craftedData?: Card;    // crafted の場合、カード定義そのものを保持
  acquiredAt: number;
}

// マテリアル種別
type MaterialCategory =
  | 'stat_hp'
  | 'stat_atk'
  | 'movement'
  | 'attack_range'
  | 'skill';

// マテリアル定義
interface Material {
  id: string;
  name: string;            // 例：「HP+2」「騎馬」「貫通×3」
  category: MaterialCategory;
  description: string;     // ツールチップ用説明
  cost: number;            // 既存付録Cの計算式に基づく
  // カテゴリ別の効果データ
  effect:
    | { type: 'stat_hp'; value: number }
    | { type: 'stat_atk'; value: number }
    | { type: 'movement'; pattern: MovementPattern }
    | { type: 'attack_range'; range: AttackRange }
    | { type: 'skill'; skill: Skill };
}

// 所持マテリアル
interface OwnedMaterial {
  materialId: string;
  count: number;
}

// デッキ
interface Deck {
  deckId: string;
  name: string;            // プレイヤー命名（最大15文字）
  cardEntries: { ownedCardId: string; count: number }[];
  createdAt: number;
  updatedAt: number;
}

// クエスト定義（ハードコードされた静的データ）
interface QuestDefinition {
  questId: string;
  chapter: number;          // 章番号
  order: number;            // 章内の順序
  title: string;
  description: string;
  // 開始時の会話シーン（任意）
  prologue?: DialogueScene[];
  // 敵デッキとAI設定
  enemyDeckId: string;
  enemyAiLevel: 'easy' | 'normal' | 'hard';
  // 特殊勝利条件（任意）
  specialRule?: 'use_only_militia' | 'win_in_5_turns' | null;
  // クリア時の会話シーン（任意）
  epilogue?: DialogueScene[];
  // 報酬
  reward: Reward;
  // 解放条件（前提クエスト）
  prerequisites: string[];   // questId の配列
}

interface DialogueScene {
  speaker: string;           // キャラクター名
  iconKey?: string;          // 立ち絵識別子
  text: string;
}

interface Reward {
  exp: number;
  runes: number;
  cards?: { cardId: string; count: number }[];
  materials?: { materialId: string; count: number }[];
}

// クエスト進行
interface QuestProgress {
  questId: string;
  status: 'locked' | 'available' | 'in_progress' | 'cleared';
  clearedAt?: number;
  attemptCount: number;
}

// ガチャ排出テーブル
interface GachaTable {
  tableId: string;
  name: string;
  pricePerPull: number;
  pullsForBundle: number;       // 例：10
  bundlePrice: number;
  rarityWeights: Record<Rarity, number>;
  guaranteedRarity?: Rarity;    // 10連の最低保証
}
```

### 3.2 Firestore スキーマ

```
users/{userId}
  ├── profile (document)
  │     { level, exp, runes, createdAt, updatedAt }
  │
  ├── inventory/cards (document)
  │     { ownedCards: OwnedCard[] }
  │     ※カード総数が膨大にならない見込みなら単一ドキュメントで保持
  │
  ├── inventory/materials (document)
  │     { ownedMaterials: OwnedMaterial[] }
  │
  ├── decks/{deckId} (collection)
  │     Deck
  │
  ├── quests/{questId} (collection)
  │     QuestProgress
  │
  └── sessions/{sessionId} (既存)
```

**設計の根拠：**
- カード・マテリアルは件数が増えても種類数は限定的（数百以内想定）→ 単一ドキュメントで保持して読み取りコスト削減
- デッキとクエストは個別操作が多い → コレクション形式で柔軟に
- 既存の `sessions/{sessionId}` は変更しない

### 3.3 Firestore セキュリティルール

```
match /users/{userId} {
  match /{document=**} {
    allow read, write: if request.auth.uid == userId;
  }
}
```

**注意：**`profile` と `inventory` はサーバー検証が必要な書き込み（ガチャ、マテリアル購入、抽出、生成、報酬付与）が多い。MVP ではクライアントから直接書き込むが、**将来 Cloud Functions に移行する前提でロジックを `src/lib/server-logic/` 等に切り出しておく**こと。

具体的には：
- `applyReward(profile, inventory, reward)` のような pure function として実装
- これをクライアントから呼んで Firestore に書き戻す
- 将来 Cloud Functions に同じ pure function を呼ばせるだけで移行できる構造

### 3.4 静的データ（リポジトリにコミットする定義データ）

以下は Firestore ではなく `src/lib/data/` に TypeScript 定数として持つ：

| ファイル | 内容 |
|---|---|
| `src/lib/data/cards.ts` | 既存 13 カードの定義（rarity 等を追加） |
| `src/lib/data/materials.ts` | マテリアル全種の定義 |
| `src/lib/data/quests.ts` | クエスト全定義 |
| `src/lib/data/dialogues.ts` | 会話シーンの定義（クエストから参照） |
| `src/lib/data/gacha.ts` | ガチャテーブル定義 |
| `src/lib/data/economy.ts` | EXP テーブル、レベル別コスト上限、価格計算ヘルパー |
| `src/lib/data/icons.ts` | アイコン識別子 → 表示要素のマッピング |

```
[/login] ── 認証
```

### 4.2 主要画面のワイヤーフレーム規約

すべて **モバイル縦持ち（375px 〜 480px 幅）** を主戦場とする。レイアウトは以下のテンプレートに従う。

```
┌────────────────────────────────┐ ← 画面幅（safe-area 考慮）
│ [共通ヘッダー：48px]           │
│  Lv 5 [████░░] 1200/2500 EXP  │
│  💰 1,840 ルーン  ⚙️           │
├────────────────────────────────┤
│                                │
│  画面ごとのメインコンテンツ      │
│  （縦スクロール）                │
│                                │
│                                │
├────────────────────────────────┤
│ [ボトムタブ or アクション領域]  │ ← 必要時のみ
└────────────────────────────────┘
```

**全画面共通：**
- 戻るボタンは画面左上の固定位置（ヘッダー左）

- 確認モーダルは画面下からスライドアップ（ボトムシート）

## 5. UI/UX 統一ルール

### 5.1 デザイン言語（既存方針の継続）

- **mecraft 由来のグラスモーフィズム**を継続：背景の半透明パネル + ぼかし + 内側ボーダー
- **メイン背景**：濃紺〜ダークブルー（`#0a0e27` 〜 `#1a1a2e`）のグラデーション
- **アクセントカラー**：
  - プレイヤー要素：青系（`#3b82f6`）
  - 敵要素：赤系（`#ef4444`）
  - 通貨・報酬：黄金（`#fbbf24`）
  - 経験値：水色（`#22d3ee`）
  - レアリティ：C=灰、R=青、SR=紫、SSR=金
- **フォント**：Tailwind デフォルト（システムフォント）。見出しは `font-bold`、本文は `font-medium`

### 5.2 共通コンポーネント（`src/components/ui/`）

実装前に以下を**先に作る**。各画面はこれらを組み合わせて構築する。

| コンポーネント | 役割 |
|---|---|
| `<AppHeader />` | Lv/EXP バー、ルーン残高、設定アイコン。全画面で表示 |
| `<BackButton />` | ヘッダー左の戻るボタン。`router.back()` |
| `<GlassPanel />` | グラスモーフィズム化した汎用カードパネル |
| `<PrimaryButton />` | 主要CTAボタン（青グラデーション、最低 48×48px） |
| `<SecondaryButton />` | サブボタン（グラスパネル風） |
| `<DangerButton />` | 不可逆操作用（赤系、確認モーダル必須） |
| `<RuneCost runes={n} />` | ルーンアイコン＋数値の表示。残高不足時は赤字 |
| `<RarityBadge rarity={r} />` | C/R/SR/SSR のバッジ |
| `<CardThumbnail card={c} size="sm|md|lg" />` | カード表示の標準化（後述） |
| `<MaterialChip material={m} count={n} />` | マテリアルの小型表示 |
| `<ConfirmSheet />` | ボトムシート型の確認モーダル（不可逆操作用） |
| `<RewardModal reward={r} />` | 報酬獲得時のモーダル（共通演出） |
| `<EmptyState />` | 「まだカードがありません」等の空状態 |

### 5.3 カード表示の統一仕様

カード表示は**3サイズ**に統一する：

| サイズ | 用途 | 寸法 |
|---|---|---|
| `sm` | デッキビルダーの一覧、コレクションのグリッド | 80×112px |
| `md` | 手札、ガチャ結果 | 120×168px |
| `lg` | 詳細モーダル、生成プレビュー | 240×336px |

カードの構成要素（どのサイズでも統一）：
```
┌─────────────────┐
│ [レアリティ枠]    │  ← 枠色 = レアリティカラー
│  [アイコン]       │  ← iconKey または絵文字
│  [カード名]       │
│ ATK 2  HP 3      │  ← サイズ md 以上で表示
│ コスト 6          │  ← サイズ sm でも表示
└─────────────────┘
```

**重要：**カード画像は当面 SVG プレースホルダー＋絵文字で実装（既存方針継続）。生成カード（crafted）の場合は `iconKey` で選んだアイコンを表示する。

### 5.4 不可逆操作のガード

以下は**必ず ConfirmSheet による2段階確認**を経由：

- カード抽出（カードが消える）
- マテリアル消費を伴うカード生成
- ガチャ実行（ルーン消費）
- マテリアル購入（ルーン消費）
- デッキ削除

確認モーダルには**消費するもの／得るものを左右対比**で表示すること。

### 5.5 報酬獲得演出（統一）

`<RewardModal />` は以下の構造で**全画面共通**にする：

```
┌────────────────┐
│  REWARD!       │  ← タイトル
│  ─────────     │
│ +30 EXP        │  ← 経験値（水色のバー演出）
│ +50 ルーン      │  ← ルーン（黄金）
│ ─────────      │
│ [カード] [カード]│  ← 獲得カード（フェードイン）
│ ─────────      │
│ [マテリアル]    │  ← 獲得マテリアル
│                │
│ [タップで閉じる]│
└────────────────┘
```

**演出原則：**
- アニメーションは合計2秒以内
- スキップ可能（タップで即座に閉じる）
- レベルアップが発生した場合、**最後にレベルアップ演出を追加**で重ねる

### 5.6 ローディング・遷移

- 画面遷移時のフェード：150ms 以下
- データ取得中のローディングはスケルトン表示（テキストプレースホルダーではなく）
- 1秒以上かかる処理は進捗テキスト（「鍛冶中...」「召喚中...」等のフレーバー）

### 5.7 エラー・空状態

- ルーン不足：ボタン無効化＋ボタン下に「ルーンが足りません」を**赤字小文字**で
- カード/マテリアル所持なし：`<EmptyState />` で**次にすべき行動への導線**を必ず提示
  - 例：「カードを持っていません」→「ガチャを引く」「ストーリーを進める」ボタン

### 5.8 タップ領域とジェスチャー

- インタラクティブ要素は **48×48px 以上**
- 隣接要素間は **8px 以上**の間隔
- 横スクロールは scroll-snap で要素単位スナップ
- ロングタップで詳細モーダルを開くのは**避ける**（モバイルで誤発火しやすい）

## 6. 機能仕様

### 6.1 プレイヤープロフィール

#### 機能
- レベル・累積EXP・ルーン残高の保持
- EXP 加算時のレベルアップ判定
- レベルアップ通知（演出）

#### 実装ポイント
- `src/lib/server-logic/profile.ts` に pure function として実装
  - `addExp(profile, amount): { profile, leveledUp, levelsGained }`
  - `addRunes(profile, amount): profile`
  - `consumeRunes(profile, amount): profile | null`（不足時は null）
- レベルアップは EXP 累積で判定。`getLevelFromExp(exp): number` をユーティリティに
- 初回プロフィール作成タイミング：認証直後、`profile` ドキュメントが存在しなければ初期値で作成

#### 初期値
```
{ level: 1, exp: 0, runes: 500 }  // ルーン500は最初のガチャ2回分
```

#### UI
- ヘッダー左：Lv 表示＋ EXP バー（次レベルまでの進捗）
- ヘッダー右：ルーンアイコン＋残高
- `/profile` 画面：詳細統計（総試合数、勝率、所持カード総数、所持マテリアル総数等）

### 6.2 ホーム画面（`/`）

#### 機能

### 6.5 マテリアル抽出（`/collection` 内の動線）

#### 機能
- 所持カードを消費して、構成マテリアルを得る
- **不可逆操作**

#### 抽出ロジック（確定事項）
- preset カード：付録Cの計算式から「逆引き」した固定マテリアル構成を返す
  - 例：民兵（移動：前1+ATK1+HP1+攻撃前1+コスト4）→ `move_forward_1, stat_atk_1, stat_hp_1, range_melee_1`
- crafted カード：そのカードを生成したときに使ったマテリアルの記録（`craftedFrom: materialId[]`）から完全に復元
- **得られるマテリアルはカードを構成した全てのマテリアル**（半端な還元はしない）

#### UI フロー
1. コレクション画面 → カード詳細モーダル
2. 「抽出する」ボタン → ConfirmSheet 表示
3. ConfirmSheet 内：
   - 左：失うもの（カード × 1枚）
   - 右：得るもの（マテリアル一覧）
4. 「実行」ボタン → 処理実行 → トースト通知「抽出完了」
5. 該当カードの所持数が 0 になった場合、コレクション一覧に戻る

#### 制約
- 所持枚数 1 のカードでも抽出可能（= デッキで使えなくなる）
- 抽出料金：**無料**（初期実装）
- 抽出回数の制限：なし

### 6.6 マテリアルショップ（`/shop`）

#### 機能
- マテリアルをルーンで購入
- 全マテリアル常時購入可能（初期実装）
- 日替わり目玉商品（任意拡張、Phase 6 以降）

#### 価格計算
```
価格 = マテリアルのコスト値 × 50 ルーン
```
例：HP+2（コスト2）→ 100 ルーン、貫通×3（コスト5）→ 250 ルーン

#### UI
- マテリアル一覧（カテゴリタブ + チップ表示）
- 各チップに価格表示、所持数表示
- タップで購入確認 ConfirmSheet
- 連続購入対応（数量選択は最初は不要、後で +1 / +5 / +10 のステッパー）

### 6.7 ガチャ（`/gacha`、UI ラベルは「召喚」）

#### 機能
- ルーン消費でランダムにカード入手
- 単発・10連の選択

#### 排出設定（初期）
```
価格：単発 200 ルーン / 10連 1800 ルーン（10%引き）
排出率：
  C   70%（民兵、軽歩兵、急襲兵、偵察兵、槍兵）
  R   22%（重装兵、戦闘兵、弓兵）
  SR   7%（衛兵、治癒兵、騎兵、大砲）
  SSR  1%（守護兵）
10連の保証：最低でも R 以上が1枚
```

#### UI フロー
1. ガチャ画面：単発／10連ボタン、所持ルーン表示
2. ボタンタップ → ConfirmSheet（価格・引く回数）
3. 実行 → ガチャ演出（裏向きカード → 表向きフリップ）
4. 結果表示（10連の場合は全カードを一覧で表示）
5. 「もう一度引く」「閉じる」ボタン

#### 演出
- 単発：1枚をズームインしてフリップ（1.5秒）
- 10連：全10枚を一覧表示後、SSR が含まれる場合は最後に大演出（金の光）
- スキップ可能（タップで即時結果表示）

### 6.8 カード生成・鍛冶（`/forge`）

> **このゲームの核機能。最も丁寧に UX を作る。**

#### 機能
- 所持マテリアルを組み合わせてオリジナルカードを生成
- プレイヤーレベルに応じたコスト上限内で構築
- 命名・アイコン選択

#### 必須要素（生成時に必ず含める）
- 移動 × 1（移動なしカードは作れない）
- 攻撃範囲 × 1（攻撃範囲なしカードは作れない、ただし `range_none` という特殊マテリアルを持つ場合は例外）
- ATK × 1個以上（合計 ATK は 0 でも可。例：治癒兵風）
- HP × 1個以上（合計 HP ≥ 1 必須）
- スキル × 0〜1（任意）

#### コスト上限テーブル（プレイヤーレベル別）

| Lv | 最大コスト | 想定で作れるもの |
|---|---|---|
| 1 | 6 | 民兵〜槍兵相当 |
| 2 | 8 | 戦闘兵相当まで |
| 3 | 10 | |
| 4 | 12 | 衛兵相当まで |
| 5 | 14 | 治癒兵・騎兵相当まで |
| 6 | 16 | |
| 7 | 18 | 大砲相当まで |
| 8 | 20 | |
| 10 | 25 | |
| 15 | 35 | |
| 20 | 50 | 守護兵超えの高度カード |

詳細は `src/lib/data/economy.ts` の `LEVEL_COST_CAP` 定数で管理。

#### UI（最重要画面・モバイルファースト）

```
┌────────────────────────────────┐
│ [← 戻る]  鍛冶  Lv5 上限14       │
├────────────────────────────────┤
│  [生成プレビューカード（lg）]    │
│  名前：[未入力]                  │
│  ATK 2  HP 3  コスト 6/14        │
├────────────────────────────────┤
│  ── スロット ──                  │
│  移動：     [騎馬（前2ジャンプ）] │
│  攻撃範囲： [近接（前1）]         │
│  ATK：      [ATK+1] [ATK+1]      │
│  HP：       [HP+2] [HP+1]        │
│  スキル：   [+追加]              │
├────────────────────────────────┤
│  [マテリアル選択シート]          │
│  カテゴリタブで選択              │
├────────────────────────────────┤
│  [カード名を入力] [アイコン選択] │
│  [生成する] (要件未満は無効)     │
└────────────────────────────────┘
```

**設計詳細：**
- スロット形式（タップで該当カテゴリのマテリアル選択シートが開く）
- ATK/HP は複数積み可能（合算される）
- リアルタイムでコスト合計、ATK/HP 合計、コスト上限との比較を表示
- コスト超過時はプレビューを赤枠表示、「生成」ボタン無効
- 必須要素未充足時もボタン無効＋下に「移動を選択してください」等のメッセージ
- カード名：最大15文字、空欄不可
- アイコン：プリセットの絵文字 / SVG から選択（10〜20種類）

#### 生成実行
1. 「生成する」タップ → ConfirmSheet：「マテリアルを消費してカードを生成します」
2. 実行 → 鍛冶演出（ハンマー音＋火花、1.5秒）
3. 完成カードのお披露目演出
4. コレクションに追加され、`/collection` または `/forge` トップに戻る

#### データ
生成カードは `instanceId` を `crafted_${userId}_${timestamp}_${rand}` で発行し、`craftedData` に Card 定義を埋め込んで `OwnedCard` として保存。

### 6.9 デッキ管理（`/deck`）

#### 機能
- 複数デッキの作成・編集・削除
- 各デッキに名前を付けられる
- 対戦・クエスト開始時にデッキ選択

#### 制約
- デッキ枚数：20枚固定（既存仕様）
- 同一カード上限：2枚（既存仕様）
- デッキ最大数：5（初期）
- 削除可能だが、最後の1個は削除不可

#### UI フロー
1. デッキ一覧画面：所有デッキのカード（コスト分布グラフ付き）
2. 「新規作成」または既存デッキタップで `/deck/edit?deckId=xxx`
3. ビルダー画面：
   - 上半分：現在のデッキ（20枚分のスロット表示）
   - 下半分：所持カード一覧（追加可能なカードが青枠、上限到達は灰色）
   - タップで追加／削除
4. 「保存」で確定、「破棄」で編集前に戻す

#### 既存対戦への組み込み
- `/play` 開始時にデッキ選択画面を経由（デッキが1つしかない場合は自動選択でスキップ）

### 6.10 ストーリー / クエスト（`/story`）

#### 機能
- 章立てされたクエスト一覧
- クエスト挑戦・クリア・報酬獲得
- 会話シーン演出

#### クエスト構造
```
Chapter 1：鍛冶師見習い（チュートリアル兼任）
  Quest 1-1 … 基本操作チュートリアル
  Quest 1-2 … スキル使用チュートリアル
  Quest 1-3 … ベース攻撃チュートリアル
  Quest 1-4 … 初の本気の対戦
  Quest 1-5 … 章のボス

Chapter 2：南の砦
  Quest 2-1 〜 2-5

Chapter 3：北の山岳
  Quest 3-1 〜 3-5

...
```

**初期実装では Chapter 1（5本）+ Chapter 2（5本）+ Chapter 3（5本）= 15本を用意**。

#### UI
- 章選択画面（横スワイプで章切替、`/story`）
- 章内のクエストはマップ風に縦に並ぶ（または単純なリスト）
- 解放済み・進行中・クリア済みの状態が一目で分かるアイコン
- クリア済みクエストは再挑戦可能（報酬は初回のみ）

#### クエスト実行フロー
1. クエスト選択 → プロローグ会話（ある場合）→ デッキ選択（チュートリアル中は固定デッキ強制）
2. 対戦画面（既存 `/play` を流用、`questId` クエリで分岐）
3. 勝利 → エピローグ会話 → 報酬演出 → クエスト一覧に戻る
4. 敗北 → リトライ／撤退の選択

#### 会話シーン UI
- 画面下部に固定の会話パネル（テキスト送りタップ）
- 立ち絵は画面中央〜上部に表示（任意）
- 背景は暗転または該当背景画像
- スキップ可能（タップで全文表示 → もう一度タップで次へ）

### 6.11 チュートリアル（クエスト Chapter 1 に統合）

#### 設計
- チュートリアルは独立した機能ではなく **Chapter 1 の Quest 1-1, 1-2, 1-3 に組み込む**
- スクリプト駆動の対戦体験：
  - Quest 1-1：召喚・移動・攻撃の基本（敵は最弱固定 AI、ハイライトと吹き出しガイド付き）
  - Quest 1-2：スキル使用（弓兵の貫通を体験）
  - Quest 1-3：ベース攻撃で勝利（敵 HP 1 状態から開始）
- **強制ガイド：**Quest 1-1 のみ、特定の操作以外を無効化
- **ガイドUI：**画面端から吹き出し、操作対象をスポットライト

#### 実装ポイント
- `src/components/tutorial/TutorialOverlay.tsx` を作る
- クエストデータに `tutorialSteps?: TutorialStep[]` を持たせる
- TutorialStep は `{ targetTestId, message, allowedActions }` で構成
- 既存の対戦画面（`/play`）にオーバーレイを重ねる

#### スキップ
- 2回目以降の起動はチュートリアル全体スキップ可能
- ただし「完了済み」状態で `quests/{questId}` に記録

### 6.12 報酬システム

#### 機能
- クエストクリア・対戦勝利時に報酬を付与
- `<RewardModal />` で統一演出

#### 通常対戦の報酬
- 勝利：+50 ルーン、+30 EXP
- 敗北：+10 ルーン、+10 EXP（参加賞）
- 引き分け：+20 ルーン、+15 EXP

#### クエスト報酬（参考値）
| クエスト | EXP | ルーン | 追加 |
|---|---|---|---|
| Quest 1-1 〜 1-3（チュートリアル） | 50 | 100 | 各カード1枚（弓兵・治癒兵等） |
| Chapter 1 通常クエスト | 100 | 200 | マテリアル数個 |
| Chapter 1 章ボス | 300 | 500 | 特定カード1枚 |
| Chapter 2 通常 | 150〜200 | 300 | マテリアル数個 |
| Chapter 2 章ボス | 500 | 800 | 特定カード1枚 |

詳細は `src/lib/data/quests.ts` で個別調整。

#### 実装
- `src/lib/server-logic/reward.ts` に pure function `applyReward(profile, inventory, reward)` を実装
- クエスト勝利・対戦勝利の両方からこの関数を呼ぶ
- 報酬付与後に `<RewardModal />` を必ず表示

### 6.13 既存対戦への統合

#### 改修ポイント（既存コード）
- `/play` 開始時：デッキ選択ステップを追加（デッキが1個ならスキップ）
- `/play` 終了時：勝敗に応じて報酬計算 → Firestore 反映 → `<RewardModal />` 表示 → `/result` へ
- 既存の `sessions/{sessionId}` は変更しないが、`deckId` を記録する

#### 互換性
- 既存の固定標準デッキ（`buildStandardDeck()`）は**初期デッキ**として残す
- 新規ユーザーには「標準デッキ」として最初から付与（プロフィール作成と同時に decks/{default} を作成）

### 7.3 通貨フロー（初期バランス）

| 入手 | 単価 |
|---|---|
| 通常対戦勝利 | 50 ルーン |
| 通常対戦敗北 | 10 ルーン |

### 7.4 バランスの懸念点（プレイテストで要検証）

- ガチャ → 抽出 → マテリアル化 → 再ガチャ のループで**ルーンが増殖する可能性**
  - 期待値検証：ガチャ単発200ルーン → 期待カードコスト 6.67 → 抽出マテリアル価値 333 ルーン → 純益 +133 ルーン
  - **対策候補**：抽出料金を導入、ガチャ価格を上げる、マテリアルショップ価格を調整、抽出時に一部マテリアルがロスト
  - **初期実装：上記初期値で実装してプレイテスト**。問題が出たらマテリアルショップ価格を `× 80` に上げる
- 高 SSR を引いた場合の即時抽出：1個の抽出で大量マテリアル → 強カード生成の連鎖
  - 当面は許容（プレイヤー報酬として）

## 10. 実装上の注意事項

### 10.1 コードの分離

- ゲームコアロジック（`src/lib/game/`）と メタゲームロジック（`src/lib/meta/`）を分離
- データ定義（`src/lib/data/`）はロジックから独立
- サーバー側に移行する可能性のある処理（`src/lib/server-logic/`）は pure function で実装

### 10.2 命名統一

- 通貨は **「ルーン（Runes）」**で統一。「コイン」「ジェム」等は使わない
- カード生成は UI 表示「鍛冶」、関数名「craft」「forge」のいずれかに統一（**`forge` を採用**）
- ガチャは UI 表示「召喚」、関数名「gacha」「summon」（**`summon` を採用**）

### 10.3 既存コードの改修方針

- 既存 `cards.ts` に rarity を追加。値は §11.4 を参照
- 既存 `decks.ts` の `buildStandardDeck()` は残し、新規ユーザーの初期デッキとして利用
- 既存 `rules.ts`, `skills.ts`, `ai.ts` は変更不要

### 10.4 国際化

- 当面は日本語のみ
- ただし文字列は `src/lib/data/strings.ts` に集約しておく（将来 i18n 導入時に楽）

### 10.5 アセット

- カード画像、立ち絵、UI アイコン
- 既存方針通り、まずは絵文字＋SVG プレースホルダーで実装
- Phase 6 完了後に `tools/gen_images.py` で画像生成

## 11. 設計判断と確定事項

実装中に判断に迷ったらここを参照する。

### 11.1 マテリアル抽出
- **抽出は無料、構成マテリアル全戻し**
- preset カードは付録Cの計算式から導出した固定マテリアル構成
- crafted カードは生成時の `craftedFrom` から完全復元
- 抽出回数制限なし

### 11.2 生成カードのアート
- アイコンはプリセットから選択（10〜20種の絵文字 or SVG）
- 背景・枠はレアリティで決定（生成カードのレアリティはコスト合計で算出）
  - コスト 〜6: C
  - コスト 7〜11: R
  - コスト 12〜17: SR
  - コスト 18+: SSR

### 11.3 生成カードの命名
- プレイヤー命名（必須・最大15文字、空欄不可）
- 重複可（プレイヤー間）
- 命名後の変更：初期実装では不可、Phase 6 以降に検討

### 11.4 既存13カードのレアリティ付与

### 11.5 チュートリアル粒度
- Chapter 1 の最初の3クエストに統合
- Quest 1-1：基本操作（強制ガイド）
- Quest 1-2：スキル使用（半強制ガイド）
- Quest 1-3：ベース攻撃（軽いガイド）
- Quest 1-4 以降：通常クエスト

### 11.6 サーバー検証
- 初期実装はクライアント完結（pure function を切り出して Cloud Functions 化容易にする）
- Phase 6 以降、不正対策が必要になったら Cloud Functions へ移行

### 11.7 オフライン対応
- 通信不能時：キャッシュからの読み取りのみ可能、書き込みはエラー表示
- 試合中の通信断：ゲーム継続可能、結果保存時に再試行
